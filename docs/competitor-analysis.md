## 2. 各プロジェクト詳細分析

### 2.1 harness-mem（自プロジェクト）

**概要**: 複数のAIコーディング環境にまたがる統一メモリランタイム＆CLIツール。Codex、OpenCode、Cursor、Claude Code向けにワンセットアップでメモリ配線を提供。

**技術スタック**:
- TypeScript / Bun / Node.js 18+
- SQLite (WAL mode) + sqlite-vec（ベクトル検索）
- MCP (Model Context Protocol)
- React 19 + Vite 7（UI）
- SSE（リアルタイムフィード）

**主要機能**:
| 機能 | 詳細 |
|------|------|
| 自動セットアップ | Codex, OpenCode, Cursor, Claude MCP の一括配線 |
| Doctor | 健全性チェック＆自動修復 |
| スモークテスト | E2Eプライバシー・検索品質チェック |
| マイグレーション | claude-mem からの移行サポート |
| ハイブリッド検索 | 語彙＋ベクトル検索（hybrid_v3） |
| プライバシーフィルタリング | デフォルトで機密データ非表示 |
| ライブフィード | SSE によるリアルタイムイベント配信 |
| セッション管理 | 作成・チェックポイント・サマリー付き終了 |
| クロスプラットフォーム取込 | 4ツールからの統一イベント取込 |

**アーキテクチャ**:
```
harness-mem/
├── memory-server/    # コアデーモン (Bun/SQLite/ベクトル)
├── mcp-server/       # MCP クロスクライアント通信
├── harness-mem-ui/   # React ダッシュボード
├── scripts/          # CLI & フックハンドラ
└── hooks/            # グローバルフック設定
```

**強み**:
- 唯一の「マルチツール統合メモリ」という明確な差別化
- ローカルファースト・プライバシーバイデフォルト
- SQLite + ローカルベクトル（外部API不要）
- 包括的なテスト（ユニット8 + 統合9 + E2E + スモーク）
- CI/CD パイプライン完備（GitHub Actions + npm自動公開）

**弱み**:
- 非常に新しい（開発2日目、21コミット）
- コミュニティ・ユーザーベースがまだゼロ
- バージョン 0.1.x（安定性未証明）
- ドキュメントは基本的な水準のみ

---

### 2.2 mem0 (mem0ai/mem0)

**概要**: AIエージェント・アシスタント向けのインテリジェントな永続メモリレイヤー。LLMの「状態を持たない」問題を解決し、セッション間でユーザー情報を記憶。

**技術スタック**:
- Python 66.4% / TypeScript 20.7%
- 15以上のベクトルDB対応（Qdrant, Pinecone, Chroma等）
- 7以上のLLMプロバイダ対応
- グラフDB対応（Neo4j, Memgraph）
- OpenAI GPT-4.1-nano（デフォルトLLM）

**主要機能**:
- マルチレベルメモリ（ユーザー/セッション/エージェント）
- セマンティック検索 + リランキング
- グラフメモリ（エンティティ関係）
- 自動メモリ抽出（LLMベース）
- メモリ履歴（変更追跡）
- 高度なメタデータフィルタリング
- Python/TypeScript デュアルSDK
- フレームワーク統合（LangChain, CrewAI等）

**強み**:
- 巨大なコミュニティ（47.5k stars, 4,600+ 依存プロジェクト）
- プラグイン可能な設計（ベンダーロックイン回避）
- シンプルで直感的なCRUD API
- ベンチマーク裏付き（LOCOMO）
- Apache 2.0（商用利用自由）

**弱み**:
- OpenAI APIキー必須（デフォルト）
- PostHog テレメトリが必須依存
- 421 open issues / 322 open PRs（メンテナ不足の兆候）
- メモリ抽出品質がLLMに依存
- TTL/有効期限機能なし

---

### 2.3 claude-mem (thedotmack/claude-mem)

**概要**: Claude Code専用の永続メモリ圧縮システム。ライフサイクルフック経由でセッション間のコンテキストを自動キャプチャ・圧縮・再注入。

**技術スタック**:
- TypeScript / Bun
- SQLite FTS5 + ChromaDB（ハイブリッド検索）
- Claude Agent SDK
- MCP (Model Context Protocol)
- React 18 + Express（Web UI）
- ONNX（ローカル埋め込み）

**主要機能**:
- ゼロ設定の自動コンテキストキャプチャ
- AI圧縮によるメモリ要約
- プログレッシブ・ディスクロージャー（~10xトークン節約）
- ハイブリッド検索（FTS5 + ベクトル）
- Webビューアー（localhost:37777）
- `<private>` タグによるプライバシー制御
- セルフヒーリング・メッセージキュー

**強み**:
- 大規模コミュニティ（28.9k stars, 1.9k forks）
- トークン効率の優れた設計（プログレッシブ・ディスクロージャー）
- ゼロ設定のUX
- 極めて高速なリリースサイクル

**弱み**:
- Claude Code専用（他ツール非対応）
- SQLite + ChromaDB + ワーカーデーモンの運用複雑性
- ChromaDB の脆弱性（Rustパニック、ONNX破損）
- Windows サポートが不安定
- 重い依存チェーン（Bun + uv + ChromaDB + SQLite + Node.js）
- AGPL-3.0 + PolyForm Noncommercial（商用利用に制約）
- データ重複問題（Issue #1158）

---

### 2.4 memos (usememos/memos)

**概要**: プライバシーファーストのセルフホスト型メモ・ナレッジ管理プラットフォーム。軽量なマイクロブログ風のメモ管理。

**技術スタック**:
- Go 54.1% / TypeScript 44.4%
- REST + gRPC デュアルプロトコル API
- SQLite / MySQL / PostgreSQL 対応
- React 18 + Vite + Tailwind CSS + Radix UI
- Protocol Buffers (Buf)

**主要機能**:
- Markdownネイティブ（GFM, KaTeX, Mermaid対応）
- タグ・メモ関連付け・コメント・リアクション
- マルチユーザー（ロールベースアクセス制御）
- OAuth/OIDC SSO
- RSS フィード + Telegram bot + Webhook
- 多言語対応（i18n）
- 位置情報タグ + 地図表示
- 単一バイナリデプロイ

**強み**:
- 最大のコミュニティ（57k stars, 370+コントリビュータ）
- Google AIP準拠のプロフェッショナルなAPI設計
- 単一バイナリの簡単デプロイ
- MIT ライセンス（最大限の自由）
- 月次リリースの安定した開発

**弱み**:
- AIコーディングツールとの統合なし
- メモリ/コンテキスト管理の概念なし
- ネイティブモバイルアプリなし
- 0.x バージョン（安定性保証なし）
- 認証周りの問題報告が多い

## 6. harness-mem の競争戦略提言

### 6.1 即座に活用すべき強み

1. **「唯一のマルチツール統合メモリ」** というポジショニングを前面に打ち出す
   - mem0 = 汎用AI、claude-mem = Claude専用、memos = メモアプリ
   - harness-mem = **AIコーディングツール横断メモリ**（唯一の存在）

2. **ローカルファースト + API不要** をクラウド依存の mem0 との明確な差別化に
   - OpenAI キー不要、外部サービス不要で動作
   - プライバシー要件の厳しい企業向けに訴求

3. **claude-mem からのマイグレーション** パスの充実
   - 既にマイグレーション機能あり → claude-mem の安定性問題を抱えるユーザーの受け皿に

### 6.2 優先的に改善すべき領域

| 優先度 | 領域 | 具体的アクション |
|--------|------|------------------|
| **P0** | コミュニティ構築 | README強化、デモ動画作成、Hacker News / Reddit投稿、Claude Code公式コミュニティへの露出 |
| **P0** | ドキュメント | APIリファレンス完備、クイックスタートガイド、アーキテクチャ図の追加 |
| **P1** | 安定性証明 | テストカバレッジ拡大、ベンチマーク公開、本番運用事例の蓄積 |
| **P1** | DX改善 | `npx harness-mem setup` のワンコマンドセットアップ体験 |
| **P2** | 機能拡張 | メモリ有効期限（TTL）、メモリ品質スコア、エクスポート機能 |
| **P2** | エコシステム | VS Code拡張、Neovim統合、GitHub Actions連携 |

### 6.3 競合から学ぶべきポイント

| 競合 | 学ぶべき点 |
|------|-----------|
| **mem0** | プラグイン可能アーキテクチャ、ベンチマーク駆動の品質証明、デュアルSDK戦略 |
| **claude-mem** | プログレッシブ・ディスクロージャーのトークン効率設計、ゼロ設定UX |
| **memos** | Google AIP準拠のAPI設計、単一バイナリデプロイ、コミュニティ運営の模範 |

### 6.4 避けるべき競合の失敗

| 競合 | 避けるべき点 |
|------|-------------|
| **mem0** | issue/PR の消化遅延 → 初期からトリアージプロセスを確立 |
| **claude-mem** | ChromaDB依存の複雑性 → harness-memはSQLite+ローカルベクトルの軽量アプローチを維持 |
| **claude-mem** | デュアルライセンスの混乱 → 明確なライセンス方針を早期に確定 |
| **memos** | 0.xが長期化 → v1.0 到達の明確なロードマップを策定 |
