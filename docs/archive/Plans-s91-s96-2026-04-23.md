### 設計方針

- **opt-in デフォルト** (`partial_finalize_enabled=false`): §89-002 (semantic dedup) が landed 前は partial summary の重複蓄積リスクがあるため既定は無効。ユーザーが config で明示 ON
- **既存 API 後方互換**: `partial` / `include_partial` はどちらも optional、未指定時は現挙動
- **scheduler は daemon プロセス内の setInterval**: 外部 cron 依存を避け、daemon 停止時は自動停止
- **embedding 負荷**: partial summary は full summary と同じ embedding 経路を通る。1 tick=5 session 上限で spike を抑える

| S95-006 | `--execute` logic fix (column-order divergence) | 実 `--execute` が dry-run 見積もりの ~95% を silent skip していた bug を修正。root cause: `INSERT OR IGNORE INTO tgt.X SELECT * FROM src.X` が positional で、plugin-scoped DB と default DB の column 順が `mem_observations` / `mem_sessions` / `mem_facts` で divergent だったため、NOT-NULL 違反で OR IGNORE が黙って drop していた。fix は tgt/src の column 名 intersection を PRAGMA table_info で動的に取得し、列名指定の `INSERT INTO tgt(c1,c2,...) FROM src.X` に書き換え。event_id が non-null かつ target に同じ event_id が既に存在する場合は追加 skip (cross-DB dedupe key)。`--execute` の per-source audit log に実 delta を記録 (`obs_new`, `sess_new`, ...)。regression test `tests/merge-plugin-scoped-dbs-execute.test.sh` を追加 (24 assertions)、source/target で意図的に column 順を ずらしたfixture で動作を保証。修正版 dry-run は 3 source で `obs_new=40010` (inline 31,825 / codex 7,498 / marketplace 687) を再確認済み。実 `--execute` の再 run は operator 判断 (S95-004)。 | cc:完了 [HEAD] |

実装メモ:
- Dedupe key: `mem_observations.id` (TEXT ULID-ish) が primary。`(platform, project, session_id, created_at, content-length)` で同一 session の diff が flagged されるが、id が異なるため保守的に全件 merge する (append-only の原則)。実測 dry-run で obs_dup = 0 なので、この false-negative は事実上 meaningful な false-positive を生んでいない。
- `mem_entities` は INTEGER AUTOINCREMENT id のため、source の `(name, entity_type)` を target 側で lookup し直して `entity_id` を re-map。`mem_observation_entities` / `mem_relations` も re-map ロジック経由で挿入。`mem_links` は `(from, to, relation)` の複合キーで dedupe (id は AUTOINCREMENT なので target 側で新規採番される)。
- read-only ATTACH を `file:PATH?mode=ro` で行い、main を `:memory:` にすることで dry-run 中は **どの DB にも `-shm` / `-wal` 変更を起こさない**。`--execute` のみ target を書込み open、1 source = 1 `BEGIN IMMEDIATE` / `COMMIT` で atomic に merge。
- 実行前に考えるべき caveat:
  1. **Vector reindex**: merge した vector は既存の onnx model で生成されているはずだが、別 model version で書かれた vector が混ざっている可能性がある。merge 後 `harness_mem_admin_reindex_vectors` を走らせて不整合を検出することを推奨。
  2. **FTS index**: `mem_observations_fts` は merge 対象外 (target 側の AFTER INSERT trigger が自動で行うはず)。ただし bulk INSERT 時に trigger が発火するか、あるいは `reindexFtsWithSegmentation()` を別途流す必要があるか、実測で確認が要る。
  3. **mem_events**: 今回の merge では events は touch していない (`mem_observations.event_id` FK は `ON DELETE SET NULL`、存在しない event_id でも obs は生きる)。events を merge するか、単に NULL に落とすかは operator 判断。
