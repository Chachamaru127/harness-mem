Harness-mem は今、どう記憶を取り出しているか

巨大化したDBを毎回ぜんぶ探すのではなく、プロジェクト単位に絞り、検索用の整理データを使います。古くなった棚は、検索を止めずに裏で軽く更新する段階まで進みました。

local-first project scoped recall auto projection refresh worker offload OpenTelemetry local only HTTP MCP default ADR / decisions aware

現在の状態

daemon は起動中。ready 応答は 0.03ms。

新規 Claude Code / Codex setup は local HTTP MCP gateway が default。

OpenTelemetry 対応済み。外部送信なしの local mode。

projection_stale 検知後、自動更新で recall_projection_v1 に復帰。

ひとことで

今の Harness-mem は、まず「今のプロジェクトに関係する記憶」だけを取りに行きます。 検索用に整えた projection が新しければ高速ルートで返し、古ければ scoped fallback で即返します。Claude Code / Codex との接続は、新規 setup では local HTTP MCP gateway を使うため、複数セッションでも入口を 1 つに寄せやすくなりました。

記憶を取り出す流れ

1

質問を受ける

例: 「Recall Runtime Architecture を思い出して」

2

範囲を絞る

まず current project。必要なら session。広域検索は明示時だけ。

3

projection を見る

検索用に整理済みの記憶。新しければ本命ルート。

4

古ければ即 fallback

本体DBから project scoped に探し、同時に棚更新を予約する。

5

次回は棚に戻る

child refresh 後は recall_projection_v1 へ復帰。

どの範囲の情報を取りに行くか

範囲 いつ使うか 今の扱い
Project 通常の recall / search。今の repo の話。 第一候補。Skills でも project 指定を原則化。
Project + Session 特定の作業セッションの続き。 より狭い。混入が少ない。
ADR / decisions 「なぜ決めたか」「方針は何か」。 decision recall object として取り出せる。
WorkGraph / Plans 作業計画、依存関係、次にやること。 通常検索に混ぜすぎず、必要な時に evidence として接続。
全体 / forensic 明示的な横断調査、デバッグ、管理操作。 通常ルートでは避ける。巨大DBで重くなりやすい。

Claude / Codex とのつながり

新規 setup

Claude Code / Codex は http://127.0.0.1:37889/mcp の local HTTP gateway に接続します。gateway は同じ local daemon と DB にだけつながります。

戻し方

うまく動かない環境では harness-mem mcp-config --transport stdio --client claude,codex --write で stdio に戻せます。記憶DBは消しません。

token は HARNESS_MEM_HOME/mcp-gateway.token に owner-only permission で保存します。client config には token の値ではなく、環境変数名や placeholder だけを書きます。Hermes は引き続き明示 opt-in です。

速度の現在地

daemon ready 0.03ms 生存確認は非常に軽い
stale 検知時 644.61ms fallback で返し、refresh を予約
refresh 後 recall 60.19ms recall_projection_v1 に復帰

計測時刻: 2026-05-24 JST。stale の初回は status=scheduled を返し、約3秒後の再確認で recall_projection_v1 に戻りました。

4つの取り出しルート

本命

recall projection

整理済みデータから高速に返す。理想状態。

近道

repeat cache

同じ scope / query の繰り返しを短時間だけ速く返す。

予備

scoped fallback

projection が古い時に project 範囲で本体DBを探し、棚更新を予約する。

管理用

forensic search

横断調査用。通常の会話では使いすぎない。

今までの問題に対して何が変わったか

以前

巨大DBで検索すると daemon 本体が SQLite scan に捕まり、health まで timeout することがありました。さらに stdio MCP frontend がセッションごとに増え、どの入口が問題か見分けにくいことがありました。

現在

重い検索は worker へ逃がし、Skills は project 指定を原則化。stale / missing projection は child process で自動更新され、Claude / Codex の新規接続は local HTTP gateway に寄ります。

今回の更新で変わったところ

検索用の棚が古くなっても、ユーザーを待たせずに復旧へ向かうようになりました。 まず fallback で答えを返し、同時に recall_projection_auto_refresh を予約します。更新作業は daemon 本体ではなく child process 側で行うため、検索中も ready 応答を守れます。

今の安定化ポイント

仕組み 何を防ぐか ユーザーから見た変化
debounce refresh 新しい記憶が連続して増えた時の更新連打。 裏の棚更新がまとめられ、daemon が重くなりにくい。
one-flight queue 複数セッションから同じ project の refresh が重なること。 並行作業でも refresh storm になりにくい。
child process refresh projection rebuild が daemon 本体を塞ぐこと。 検索中も health / ready が軽いまま。
OpenTelemetry local 検索が遅い時に、どこで詰まったか分からなくなること。 Claude / Codex どちらの利用でも、daemon 側で recall latency や fallback を観測できる。
HTTP MCP gateway 複数セッションで MCP frontend が増え、切り分けが難しくなること。 新規 Claude / Codex setup では入口が local gateway に寄り、stdio へ戻す道も残る。
safe meta raw query / project path / secret が telemetry に出ること。 状態だけ分かり、内容は漏れない。

結論

今の Harness-mem は、巨大DBをそのまま毎回探す段階から、project scope、projection、fallback、worker、telemetry、自動 refresh、local HTTP MCP gateway を使う Recall Runtime に進化しています。通常の複数セッション運用はかなり安定し、検索棚が古くなっても自動で回復へ向かう状態になりました。
