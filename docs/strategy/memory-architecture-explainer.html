④ 他プロダクト／選択肢で LLM を足す「3箇所」

取り出し精度を上げたいとき、LLM を入れられる場所は3つ。harness-mem が今どうしているかを併記。

入れる場所何をするharness-mem の今足す価値
① クエリ書き換え質問を LLM が言い換え・日英展開してから検索ルールベースのみ(LLMなし)日英混在クエリに効く。狙い目
② LLM リランク上位k件だけ LLM で「本当に関係あるか」並べ替え専用 reranker(非生成)精度↑。ただし遅くなる。top-k限定なら可
③ 回答合成(RAG)上位を読んで LLM が答えを作文やらない呼ぶ側が既に LLM。二重になるので不要

⑤ いちばんの肝：埋め込みは「保存時に固定」される

あなたの問い2への答え：その通り。だから「再embedding」では直さない。

埋め込み(ベクトル)は保存した瞬間に作られ、その後変わらない。 だから後で事実が覆っても、古い記憶のベクトルは「古い意味のまま」検索に当たり続ける。 全部を作り直す(再embedding)のは重すぎる。

ベクトル(埋め込み)の役割
候補を見つける係。保存時に固定。時制では更新しない
グラフ(関係)の役割
どれが「今」か決める係。superseded + valid_to で後からいつでも更新できる

→ 鮮度は埋め込みの領域ではなく、関係グラフ(bi-temporal)の領域。あなたの直感どおり。 具体的には3手で解く：
① superseded に valid_to を書く (154-302b) ② 時制が覆ったら新 observation を追加し旧を失効 (154-303 dreaming) ③ anchor retrieval で「前の状態」も引けるように (154-302)
