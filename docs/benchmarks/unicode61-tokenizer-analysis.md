## 概要

FTS5 `unicode61` tokenizer の日本語処理動作と、`core-utils.ts` の `tokenize` 関数との整合性を検証した結果を記録する。

---

## 検証結果

### 1. unicode61 tokenizer の日本語処理

**結論: unicode61 は日本語テキストをスペース区切りで処理する。文字単位の分割は行わない。**

| テキスト | unicode61 のトークン |
|---------|---------------------|
| `"デプロイ作業が完了しました"` | `"デプロイ作業が完了しました"` (1トークン) |
| `"認証バグが発生しています"` | `"認証バグが発生しています"` (1トークン) |
| `"データベースの接続エラー"` | `"データベースの接続エラー"` (1トークン) |
| `"デプロイ 作業 が 完了"` (スペース区切り) | `"デプロイ"`, `"作業"`, `"が"`, `"完了"` (4トークン) |

**重要な含意**: スペースなしの日本語文は「1つのトークン」として扱われるため、部分一致検索が機能しない。

### 2. 日本語FTS検索の実態

```
クエリ「デプロイ」→ 0件（"デプロイ作業が完了しました" は1トークンなので不一致）
クエリ「認証」    → 0件（"認証バグが発生しています" は1トークンなので不一致）
クエリ「データベース」→ 0件（同上）
```

**スペース区切り済みテキストではヒット**:
```
"デプロイ 作業 が 完了 しました" → クエリ「デプロイ」でヒット
"認証 バグ が 発生 しています"   → クエリ「認証」でヒット
```

### 3. core-utils.ts の tokenize 関数との整合性

`tokenize()` 関数（L172-L179）の動作:

```typescript
function tokenize(text: string): string[] {
  return text
    .toLowerCase()
    .replace(/[^a-z0-9\u3040-\u30ff\u3400-\u9fff\s]/g, " ")
    .split(/\s+/)  // スペース区切りのみ
    .filter((token) => token.length > 1)
    .slice(0, 4096);
}
```

**問題**: `tokenize()` もスペース区切りのみ。日本語テキスト内でスペースがなければ全体が1トークン。

| テキスト | tokenize() 結果 |
|---------|----------------|
| `"デプロイ作業が完了しました"` | `["デプロイ作業が完了しました"]` (1要素) |
| `"認証バグが発生しています"` | `["認証バグが発生しています"]` (1要素) |
| `"データベースの接続エラー"` | `["データベースの接続エラー"]` (1要素) |

→ `tokenize()` と FTS5 tokenizer は**一致している**（両方ともスペース依存）。

### 4. SYNONYM_MAP のキーとして日本語が機能するか

`SYNONYM_MAP` のキーは `tokenize()` で正規化されたトークンと照合される。
日本語トークンはスペース区切りされた場合のみ機能する。

**例**:
- 「デプロイ データベース」→ tokenize → `["デプロイ", "データベース"]` → SYNONYM_MAP ルックアップ可能
- 「デプロイデータベース」→ tokenize → `["デプロイデータベース"]` → SYNONYM_MAP ルックアップ不可

### 5. trigram tokenizer との比較

`trigram` tokenizer を使用した場合、CJK文字を3文字単位で分割する:

| クエリ | trigram 結果 |
|--------|-------------|
| `デプロイ` | ヒット (t1) |
| `認証バグ` | ヒット (t2) |
| `データベース` | ヒット (t3) |

## 判定: Phase 2 の実現可否

### 日英SYNONYM_MAPの拡張（BM-008）

**判定: 条件付き可能**

- 日本語キーの追加は技術的に可能
- ただし、クエリにスペースが含まれている場合のみ機能する
- 例: `"deploy": ["デプロイ"]` を追加すれば「deploy」→「デプロイ」の拡張ができるが、「デプロイを実施」のようなスペースなし文では機能しない

### 日本語FTS検索の根本的な問題

現在の `unicode61` tokenizer ではスペースなし日本語テキストは検索できない。
対策として2つのアプローチがある:

**アプローチ A: trigram tokenizer への移行**
- メリット: CJK文字を自動的にN-gramで処理
- デメリット: インデックスサイズが約3-5倍になる
- 対応: `tokenize = 'trigram case_sensitive 0'`

**アプローチ B: 保存時に形態素解析してスペース区切りで格納**
- メリット: 既存インフラを維持
- デメリット: 外部形態素解析器（kuromoji等）が必要

**推奨**: BM-010（`buildFtsQuery` の日本語トークン処理改善）では、`tokenize` 関数をbigramに対応させることで**クエリ側**の改善を行う。格納側はスペース区切りのまま維持。

## 改善提案（BM-008, BM-010 向け）

### BM-008: SYNONYM_MAP 日英エントリ

追加するエントリ例（英語→日本語のマッピング）:

```typescript
// 英語キーで日本語スペース区切り同義語を追加
"deploy": ["deployment", "release", "デプロイ"],
"database": ["db", "sqlite", "データベース"],
"auth": ["authentication", "login", "認証"],
"error": ["bug", "exception", "failure", "エラー"],
"bug": ["error", "issue", "defect", "バグ"],
```

ただし、日本語クエリ → 英語同義語は `tokenize()` がスペース分割した場合のみ機能する。

### BM-010: buildFtsQuery の改善

日本語テキストの検索改善には、2文字以上のCJK文字のみを抽出するbigramロジックの追加を検討:

```typescript
// CJKシーケンスをバイグラム化する補助関数
function cjkBigrams(text: string): string[] {
  const cjkPattern = /[\u3040-\u30ff\u3400-\u9fff]+/g;
  const bigrams: string[] = [];
  for (const match of text.matchAll(cjkPattern)) {
    const chars = [...match[0]];
    for (let i = 0; i < chars.length - 1; i++) {
      bigrams.push(chars[i] + chars[i + 1]);
    }
    // 2文字以上のシーケンスはそのままも追加
    if (chars.length >= 2) {
      bigrams.push(match[0]);
    }
  }
  return bigrams;
}
```
