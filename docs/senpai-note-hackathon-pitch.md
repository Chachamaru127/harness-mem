# Senpai Note — AI Ops Hackathon 発表構成

対象: AI Ops Hackathon @関西（2026-05-31, 大阪 Blooming Camp）
テーマ: 普段の効率化しきれていない業務を AI Agent で自動化
配点: 実用性 40 / AI活用度 30 / 完成度・プレゼン 30
発表枠: 全体約60分（チーム数で割ると 1 チームおおよそ 3〜5 分想定）。本構成は
**3 分版**を基準にし、90 秒に圧縮できる短縮ガイドも併記する。

> 一言ピッチ: **「その人しか分からない業務を、誰でも 30 秒で引き継げる
> カード＋手順＋AI Agent 指示書に、Claude Code がその場で変換する。」**

---

## スライド構成（3 分・全6枚）

### S1. タイトル / 一言価値（10秒）
- Senpai Note — 業務の属人化を、引き継ぎパックに変える AI Agent
- 「作業ログ → HANDOFF_CARD / RUNBOOK / REPLAY_PROMPT」
- 採点狙い: 第一印象で「何の業務課題か」を即伝える（完成度・プレゼン）

### S2. 課題（30秒）— 実用性の布石
- 「担当者が休む / 辞める / 異動するたびに、業務が止まる・聞き直しが発生する」
- 引き継ぎは口頭やSlackやNotionに散らばり、次の人は同じ調査・同じ失敗・同じ質問を繰り返す
- 一文: **属人化は“情報が無い”のではなく、“次の人が動ける形になっていない”問題**
- 採点狙い: 実用性40 = 誰の・どの業務の・いくらの損失か

### S3. 解決策（30秒）— AI活用度の布石
- 雑な作業ログを入力するだけで、Claude Code の Skill が 3 点セットを生成:
  - **HANDOFF_CARD**: 30秒で読む引き継ぎカード（開始点・結論・やってはいけない事・次の一手）
  - **RUNBOOK**: 同種業務の再利用手順
  - **REPLAY_PROMPT**: 次の担当者の AI Agent にそのまま貼れる指示書
- ポイント: **出力がそのまま次の自動化の入力になる**（replay prompt → AI Agent）
- 採点狙い: AI活用度30 = Claude Code の Skill 機能を“業務の型”に使っている

### S4. ライブデモ（60秒）— 完成度の核
- `examples/senpai-note/ops-demo-session.md`（返金対応の雑メモ）を見せる
- Claude Code に「**このセッションを Senpai Note にして。次の人が30秒で引き継げる形に**」
- 生成された HANDOFF_CARD の「Do not repeat（注文番号を再度聞かない／未承認で返金しない）」を指差す
- REPLAY_PROMPT を別の AI に貼り、顧客返信の下書きが出るところまで見せる
- 採点狙い: 完成度30 = 実際に動く / プレゼンが明快

### S5. なぜ効くか（30秒）— 差別化と信頼性
- **単なる要約ではない**: 「次の人が動けること」に最適化（開始点・禁止事項・復旧手順）
- **source 明示**: どの記憶/ログから作ったかを必ず表示 → 嘘をつかない
- **ローカル完結 / 記憶daemon不要**: 貼ったログだけでも動くので、現場で今日から使える
- 採点狙い: 実用性40 + AI活用度30 = 安全性・現場導入性

### S6. 横展開 / クローズ（20秒）
- 経理の月次締め、サポート、採用、問い合わせ…**属人化する全業務に同じ型**
- 「作業ログを、チームの“先輩の知恵”資産に変える」
- （任意）将来: 生成 runbook を手順メモリに蓄積 / チーム共有

## 90秒短縮版（質疑が短い/枠が小さい場合）

  S6 クローズ 1 文（10秒）。スライド S3 は口頭で 1 文に統合。

## デモが詰まった時のフォールバック
- ネット/記憶daemonに依存しない: 入力は `ops-demo-session.md`、想定出力は
  `ops-handoff-pack.md` に用意済み。ライブ生成が失敗しても完成形を即提示できる。
- スクショ/印刷を 1 枚持っておく（HANDOFF_CARD の「Do not repeat」が一番刺さる）。

## 質疑想定（Q&A）
- Q: ただの要約と何が違う? → A: 開始点・禁止事項・復旧手順・次の一手まで“動ける形”で出す。Quality bar で「少なくとも1つの失敗を防ぐ」ことを必須化。
- Q: 嘘を生成しない? → A: 必ず `source:` を出し、記憶が無い時は user-provided と明記。根拠の無い断定をしない設計。
- Q: 非エンジニアでも使える? → A: 入力は雑メモでOK、記憶daemon不要。トリガーフレーズを話すだけで発火。
- Q: 既存ツール（Notion等）との違い? → A: 保管ではなく“次の人が動ける形への変換”。出力が AI Agent の指示書になる点が新しい。
